## Overview

This repository contains my solution to a Sales Effectiveness business case completed as the final stage of a recruitment process after three interview rounds.

The assessment consisted of preparing a strategic proposal and presenting it to a panel during a 20-minute presentation, followed by a discussion on the underlying reasoning behind it.

Rather than approaching the problem from a purely operational perspective, I designed an analytical framework aimed at understanding how complex business systems should be analyzed before proposing solutions.

The objective was not simply to recommend initiatives, but to demonstrate a structured methodology for diagnosing organizational problems, identifying root causes and prioritizing improvements.

# Core Idea

One of the main conclusions I reached while working on this case is that organizations often jump directly into solutions.

Instead, I believe business problems should be approached in four consecutive steps:

1.  Define the system.
2.  Define what an ideal system looks like.
3.  Understand why that ideal breaks.
4.  Only then propose improvements.

This repository documents that reasoning process.

# Analytical Framework

The framework is based on a simple assumption:

> Operational performance is not the result of isolated activities, but of the quality of the transitions between interconnected stages.

Therefore, before analyzing KPIs, I first defined the system itself.

The Lead-to-Cash pipeline was treated as an interconnected business system whose performance depends on three organizational capabilities:

- Data Governance
- Process Standardization
- Scalable Operations

These capabilities became the reference model against which every observation was evaluated.

# Diagnostic Methodology

Once the desired state was defined, I identified the organizational dimensions capable of generating vulnerabilities.

Instead of classifying problems by department, I mapped them into four root-cause dimensions:

- People
- Processes
- Technology
- Data

Departmental feedback, KPIs and operational bottlenecks were interpreted as symptoms of failures inside these dimensions rather than isolated issues.

# Decision Making

The final recommendation was not selected because it solved the largest individual problem.

Instead, initiatives were evaluated according to a single question:

> Which initiative strengthens the greatest number of foundational organizational capabilities simultaneously?

This shifted the discussion from solving symptoms to reinforcing the structural health of the system.

# Roadmap

The implementation roadmap deliberately separates foundation-building from automation.

### Phase 1

Build consistency.

- Data governance
- Standardized processes
- Approval rules
- Data validation
- Clear ownership

Only after these foundations are established does automation become valuable.

### Phase 2

Scale the process.

- Quote Generation
- Order Generation
- Agile iterations
- User feedback
- KPI monitoring

The guiding principle throughout the roadmap is:

> **"I will not automate an inconsistent process."**

# Key Takeaways

This project is less about Sales Effectiveness itself and more about a repeatable way of approaching complex business problems.

# Although the context is a commercial pipeline, the framework can be generalized to many transformation initiatives involving operations, process improvement or digital transformation.
